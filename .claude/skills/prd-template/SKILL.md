# PRD テンプレート

## トリガー

`backlog/` に新しい PRD を作成するとき。

## アクション

### 0. バッチ化確認

PRD の対象項目を決めたら、`TODO` を確認し、同じコードパス・同じ抽象パターンの項目をまとめて PRD 化すべきか検討する。

### 1. Discovery

PRD を書き始める前に、以下を行う:

1. 最低 2 つの明確化質問をユーザーに投げる
2. 回答を得てから PRD を起草する

### 2. PRD 起草

```markdown
# <タイトル>

## 背景・動機
## ゴール
## スコープ
### 対象
### 対象外
## 設計
### 技術的アプローチ
### 影響範囲
## 作業ステップ
## テスト計画
## 完了条件
```

## 設計判断の原則

- **理想的な実装を優先する**
- **一貫性**: 型システム・アーキテクチャとして一貫した解法を選ぶ
- **スコープ判断**: コストはスコープ判断の基準にならない

## 禁止事項

- Discovery をスキップして PRD を書くこと
- 曖昧な完了条件を書くこと
- 「TypeScript で表現できない」を理由に対象外とすること
- 「工数が大きい」を理由にスコープを狭めること
