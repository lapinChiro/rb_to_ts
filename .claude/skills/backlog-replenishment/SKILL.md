# backlog の補充

## トリガー

`backlog/` が空の状態でユーザーから作業依頼を受けたとき。

## アクション

1. `TODO` を確認する
2. 全項目について PRD 化の可否を判定する
3. 保留理由がある項目について、解消の可能性を自分で評価する
4. 自分で判断できない保留理由がある項目について、1 つずつ順番にユーザーに確認する
5. PRD テンプレート (/prd-template) に従い、Discovery → PRD 起草の順で進める
6. 作成した PRD を `backlog/` に配置し、`TODO` から該当項目を削除する
7. `plan.md` の消化順序に新アイテムを挿入する

### PRD 化の可否判定基準

**PRD 化可能:**
- 保留理由が明記されていない項目
- 保留理由が「設計が必要」「調査が必要」「判断が必要」のみの項目

**PRD 化不可 (正当な保留理由):**
- 別の機能・PRD が前提条件として必要
- 実際の運用データ・実績が必要
- 外部の意思決定待ち

## 禁止事項

- Discovery (明確化質問) をスキップして PRD を書くこと
- PRD 作成後に `TODO` から該当項目を削除し忘れること
- 「設計が必要」「調査が必要」を保留理由として扱うこと
