# TODO の棚卸し

## トリガー

- ユーザーから TODO の整理を依頼されたとき
- 大きな機能追加の完了後
- backlog が空で次の作業を選定する前

## アクション

### ステップ 1: 保留項目の精査

`## 保留` セクションの全項目について、保留理由がまだ有効かを確認する。

### ステップ 2: スコープ外の再評価

`## スコープ外` セクションの全項目について、前提の変化・需要の発生・工数の変化・変換可能性の再検討の観点で再評価する。

「TypeScript で表現できない」を理由にスコープ外としている項目について、本当に変換方法がないか再検討する。

### ステップ 3: PRD 化可能項目の精査

- **インスタンス数の検証**: abbrev 変換を実行し、TODO 記載のインスタンス数を照合する
- **記載内容の正確性**: 説明が現在のコードベースの状態と一致しているか
- **重複の統合**: 同じ根本原因を共有する項目のバッチ化を検討する
- **完了済み項目の除去**

### ステップ 4: スキルフィードバックの処理

`[skill-feedback:<スキル名>]` タグ付き TODO 項目を確認し、パターンを分析してスキルの修正案を提案する。

### ステップ 5: 優先順位の再評価

3 つの軸 (直接的な価値、相乗効果、伝播防止) で全項目を再評価する。

## 禁止事項

- ステップ 1〜4 をスキップしてステップ 5 だけ実行すること
- 保留理由の有効性を確認せずに「まだ有効」と判断すること
- 優先順位の判断根拠を省略すること
