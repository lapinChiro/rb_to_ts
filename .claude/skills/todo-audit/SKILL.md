# TODO 記載漏れの監査

## トリガー

1 回の開発セッション (1 つ以上のタスク完了) の最後。/quality-check, /refactoring-check の後に実行する。

## アクション

以下の 3 ステップで TODO に記載すべき未記載の課題を検出し、記録する。

### ステップ 1: コード内マーカーの確認

変更したファイルおよび影響範囲のファイルを対象に、以下のパターンを検索する:

```bash
grep -rn "TODO\|FIXME\|HACK\|WORKAROUND\|XXX" src/ tests/ --include="*.ts"
```

検出された各マーカーについて:
- `TODO` ファイルに対応する項目が存在するか確認する
- 存在しなければ `TODO` に追加する

### ステップ 2: 暫定実装の確認

今回の変更で以下のパターンが導入されていないか確認する:

- **型安全性の回避**: `as any`, `as unknown as`, `@ts-ignore`, `@ts-expect-error`
- **情報の消失**: 入力の一部を無視している
- **ハードコードされた回避策**: 特定のケースを特別扱いするコード
- **不完全なエラーハンドリング**: エラーを握りつぶしている
- **仮実装**: ダミーデータ、スタブ、プレースホルダー

### ステップ 3: 発見した課題の記録

新たに発見した課題を `TODO` に記録する:
- 記載時は「何が問題か」「なぜ問題か」「影響範囲」を具体的に書く

## 禁止事項

- 「特に問題なし」と結論する前に、ステップ 1〜2 の検索を実行せずに判断すること
- コード内の TODO コメントを `TODO` ファイルに転記せずに放置すること
- 暫定実装を発見しても「動いているから問題ない」と判断すること

## 検証

- 検索を実行した形跡がある
- 今回の変更で導入された暫定実装が全て `TODO` に記載されている
