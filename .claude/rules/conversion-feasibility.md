# 変換可能性の判断基準

## 適用条件

Ruby の構文・パターンについて「自動変換できない」「TypeScript で表現できない」と判断しようとする場面すべて。

## 制約

- 変換可能性の判断は **構文の一致** ではなく **目的の一致** で行う。Ruby のある構文が TypeScript に直接対応する構文を持たないことは、自動変換が不可能であることを意味しない
- 「TypeScript で表現できない」と結論する前に、以下の 3 ステップを必ず踏む:
  1. **目的の特定**: その Ruby コードは何を達成しようとしているか
  2. **TypeScript での代替手段の調査**: 同じ目的を達成する TypeScript のイディオム・パターンを調査する
  3. **変換戦略の設計**: Ruby の構文から TypeScript の代替手段への変換パスを具体的に設計する
- 3 ステップを踏んだ上でなお変換戦略が見つからない場合のみ「現時点では自動変換の方法が見つかっていない」と記録する。「不可能」とは書かない
- Ruby のメタプログラミング (`method_missing`, `define_method` 等) についても、まず目的を分析してから TypeScript の代替手段を検討する

## 禁止事項

- 「TypeScript の型システムで表現できない」「自動変換は不可能」と断定すること
- 構文レベルの対応がないことを根拠に変換不可と判断すること
- 目的の分析をせずに「複雑だから無理」と結論すること
